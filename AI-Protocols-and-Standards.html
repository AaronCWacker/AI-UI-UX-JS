// 01 · MCP

Model Context Protocol — tool + data layer
LAYER 1 SHIPPED · ANTHROPIC

What it is: An open standard for connecting LLMs to tools and data sources through a uniform server interface. Think USB-C for AI capabilities.

// PRIMITIVES

  • Tools — functions the model can invoke
  • Resources — read-only data the model can fetch
  • Prompts — server-side prompt templates
  • Sampling — server requesting LLM completion

// SECURITY BOUNDARY

MCP is where the model meets the outside world. Every tool call is a potential exfiltration vector. The boundary enforces: scoped credentials, per-tool consent, and no implicit data egress. A compromised MCP server can poison context — supply chain matters.

// WHY IT MATTERS

Before MCP, every app re-implemented tool calling. MCP turns the M×N integration problem into M+N. One server, many models. One model, many servers.

// 02 · A2A

Agent-to-Agent Protocol — delegation layer
LAYER 2 SHIPPED · GOOGLE + 50+ PARTNERS

What it is: A protocol for agents to discover, negotiate with, and delegate tasks to other agents across organizational boundaries — without sharing internal tools or memory.

// AGENT CARD = OPERATING CONTRACT

Each agent publishes an agent-card.json at a well-known URL declaring:

  • Identity — name, provider, version
  • Skills — what tasks it can perform
  • Inputs/Outputs — schemas it accepts and emits
  • Auth — OAuth, API key, mTLS requirements
  • Trust — endpoints, signing keys, SLAs

This is the equivalent of OpenAPI for autonomous agents. No card, no delegation.

// WHY IT MATTERS

MCP lets a model use a tool. A2A lets an agent hire another agent. The distinction matters: workers can have their own MCP servers, memory, and reasoning loops without exposing them.

// 03 · AG-UI

Agent-User Interaction Protocol — supervision layer
LAYER 3 SHIPPED · COPILOTKIT

What it is: A streaming event protocol that lets agents render real-time progress, request approvals, and surface state to a human supervisor — without the UI having to know which agent framework is running underneath.

// WHY AGENTS NEED SUPERVISION SURFACES

An agent that runs for ten minutes silently and then sends an email is a liability. Operators need to:

  • See what the agent is doing token-by-token
  • Pause before destructive actions
  • Approve tool calls above a threshold
  • Audit every step after the fact

AG-UI standardizes the event stream so this surface is portable.

// EVENT TYPES

RUN_STARTED · TEXT_MESSAGE_DELTA · TOOL_CALL_START · TOOL_CALL_END · STATE_SNAPSHOT · USER_INPUT_REQUEST · RUN_FINISHED

// WHY IT MATTERS

AG-UI is what turns "the agent did something" into "the agent did something and I watched it." It's the missing trust primitive for production agentic deployment, especially in regulated domains like healthcare.

// 04 · A2UI

Agent-to-UI — generative interface protocol
LAYER 4 CONTESTED · EARLY DRAFT

What it is: A protocol for agents to generate the UI components a user interacts with — not just stream text, but emit cards, forms, charts, and choices the host shell renders safely.

// WHY IT EXISTS

Chat is a degenerate UI. When an agent helps you book a flight, the right surface isn't a paragraph — it's a comparison card with three options, sortable by price and time. A2UI lets agents declaratively emit such surfaces.

// THE CONTESTED PART

Three open questions divide vendors:

  • Schema — JSON Schema? AdaptiveCards? Custom DSL?
  • Trust — can agents emit arbitrary HTML? (No.) What's safe?
  • Action binding — how do form submits route back into A2A?

OpenAI's Apps SDK, Google's Generative UI, and the Vercel AI SDK all stake different positions. Convergence is not yet real.

If A2UI converges on a standard, you get portable agent experiences — the same booking agent renders correctly in Slack, on iOS, in a kiosk. If it doesn't, every host re-implements its own component library. The race is on.

// 05 · AP2

Agent Payments Protocol — the mandate mechanic
LAYER 5 CONTESTED · GOOGLE + MERCHANTS

What it is: A cryptographically signed authorization framework letting users grant agents permission to spend money on their behalf, with bounded scope.

// THE MANDATE MECHANIC

A mandate is a signed envelope answering four questions:

  • Who — which user, which agent
  • What — which merchant, which SKUs, or which category
  • How much — per-transaction cap and aggregate cap
  • When — expiry window

AP2 splits this into two layers:

  • Intent Mandate — "I want to buy something matching X" (broad)
  • Cart Mandate — "Pay $47.99 to merchant Y for these items" (specific)

The agent searches under the intent mandate, then surfaces the cart mandate for human signature before checkout.

// WHY IT MATTERS

Without mandates, agents either can't pay (limit value) or hold raw card credentials (limit safety). The mandate mechanic creates a middle path: bounded, auditable, revocable spending authority.

// 06 · x402

HTTP 402 Payment Required — revived for agents
LAYER 6 CONTESTED · COINBASE + STRIPE

What it is: A re-activation of the long-dormant HTTP 402 Payment Required status code as a machine-readable payment challenge — letting any API charge agents per-call without prior account setup.

// THE FLOW IN ONE BREATH

Agent calls API → server responds 402 with payment instructions → agent pays via attached wallet → retries with proof-of-payment header → server returns the resource.

// STRIPE'S OBSERVATION

Stripe's customer-obsessed payment design philosophy says: friction in payments is friction in commerce. The historical lesson is that the simplest checkout wins — one-click, saved cards, Apple Pay. x402 extends that logic to machines: an agent shouldn't need a sign-up form to consume an API for one cent.

// COMPETING APPROACHES

  • x402 (Coinbase) — on-chain settlement via stablecoins
  • Stripe + AP2 — card rails with mandate verification
  • Hybrid — x402 envelope, any settlement underneath
